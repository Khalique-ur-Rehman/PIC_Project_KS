1. Introduction

1.1 Background

Organisations increasingly rely on cloud platforms to store and process personal data. While cloud computing provides scalability and cost savings, it also spreads information across multiple providers and regions. This raises difficult questions about who owns the data, how consent is captured, and who is accountable when things go wrong.

1.2 Project Overview

This project implements a small Cloud Data Governance Dashboard in Google Colab. The notebook generates a realistic synthetic dataset of cloud‑stored user records, runs a series of governance checks, and visualises the results. The goal is to make abstract concepts like data ownership, consent and accountability visible and measurable for students of Professional Issues in Computing.

Important Note

All records in this project are synthetic. No real personal data is used, which keeps the exercise ethical while still reflecting real governance challenges.

2. Problem Statement

When an organisation adopts cloud services, it often loses a clear view of:

  • When user data is processed for marketing or third‑party sharing without valid consent.
  • Where user data is physically stored, and whether it crosses borders into different legal jurisdictions.
  • How long different categories of data are retained compared to internal policy and legal requirements.
  • Whether sensitive information such as financial, health or biometric data is encrypted and adequately protected.

Without structured monitoring, these blind spots can lead to privacy violations, non‑compliance with regulations like GDPR, and a failure to uphold professional codes of ethics in computing.

3. Purpose & Objectives

3.1 Purpose

The project aims to provide a simple, hands‑on demonstration of cloud data governance that students can understand and extend. Instead of reading about consent, data localisation and retention in theory, learners see these ideas expressed as concrete metrics and visual dashboards.

3.2 Objectives

  • Simulate a realistic cloud‑data scenario using synthetic user records.
  • Evaluate consent, storage region, retention and encryption for each record.
  • Present the results in clear visual form using interactive graphs.
  • Map each technical finding to relevant professional, ethical and legal principles.

4. Solution Overview

4.1 Colab Notebook Flow

The core of the project is a Google Colab notebook that follows this sequence:

  1. Install and import analysis and visualisation libraries (pandas, NumPy, Plotly).
  2. Generate a dataset of around 1,000 synthetic users with attributes relevant to data governance.
  3. Run filters and counts to compute key risk metrics.
  4. Build interactive charts that summarise those metrics.
  5. Annotate each result with explanations from a professional‑issues perspective.

4.2 Dataset Design

Each row in the dataset represents one user's data stored in the cloud. Important columns include:

Identity & Location

user_id, user_country (EU, UK, US, India)

Data Type

data_category (basic, financial, health, biometric)

Cloud Region

storage_region (EU, US, Asia) to study cross‑border flows

Purpose & Consent

purpose (service, marketing, analytics, third_party_share) and consent_given

Retention

retention_days vs policy_max_retention_days

Security

encryption_at_rest flag for sensitive categories

These attributes are sufficient to explore typical governance issues such as consent violations, cross‑border storage, over‑retention and weak protection of sensitive data.

5. Link to Professional Issues in Computing

5.1 Ethics & Privacy

Professional codes from bodies such as ACM, IEEE and BCS emphasise respect for privacy, avoidance of harm and transparency with users. The dashboard directly measures:

  • How often data is used for marketing or third‑party sharing without consent.
  • Whether especially sensitive categories are handled with extra care.

5.2 Legal Compliance

Many dataset checks map to specific legal principles, particularly from GDPR:

  • Lawfulness & Consent: processing without consent risks being unlawful.
  • Storage Limitation: records exceeding their retention policy may breach this principle.
  • Data Transfers: EU users stored outside the EU raise cross‑border compliance issues.
  • Integrity & Confidentiality: unencrypted sensitive data contradicts security obligations.

5.3 Professional Responsibility in the Cloud

Cloud providers supply the underlying infrastructure, but professionals inside the organisation remain responsible for configuration and governance. The dashboard reminds engineers that they must:

  • Select appropriate cloud regions and understand jurisdictional impact.
  • Define and enforce retention rules for different categories of data.
  • Implement encryption and access controls for sensitive records.
  • Design consent flows that are understandable and genuinely informed.

6. What Each Query and Visualisation Shows

6.1 Numeric Governance Metrics

The notebook computes four key numeric indicators:

  • Marketing / Third‑Party Without Consent – counts how many records are used for marketing or partner sharing even though consent is missing. Risk: High
  • EU Users Stored Outside EU – shows how many EU residents have data stored in non‑EU regions, highlighting cross‑border and sovereignty issues. Risk: Medium–High
  • Records Exceeding Retention Policy – identifies data that is kept for longer than the organisation’s own policy allows, indicating weak deletion routines. Risk: Medium
  • Sensitive Data Not Encrypted – counts financial, health and biometric records stored without encryption; a major red flag for security and ethics. Risk: Very High

6.2 Visual Dashboards

Interactive charts turn these metrics into an easy‑to‑read dashboard:

  • Consent by Data Category (Bar Chart) – compares consent vs no‑consent for basic, financial, health and biometric data, revealing where sensitive information lacks approval.
  • User Country vs Storage Region (Heatmap) – visualises data flows between user locations and cloud regions, making cross‑border transfers immediately obvious.
  • Purpose of Processing (Pie Chart) – shows the share of data used for core service vs marketing, analytics and third‑party sharing, raising questions about data minimisation and fairness.
  • Over‑Retention by Data Category (Bar Chart) – displays which data types dominate among records that exceed retention limits; large sensitive segments signal serious governance failures.
Interpretation Tip

The numbers themselves are not from a real company, but the patterns they reveal mirror the kinds of issues a real data protection officer or cloud architect must monitor in production systems.

7. Conclusion & Future Work

7.1 Summary

The Cloud Data Governance Dashboard demonstrates how technical skills (Python, pandas, visualisation libraries) can be combined with ethical and legal thinking to analyse cloud data practices. By simulating user records and examining consent, storage, retention and encryption, the project shows where an organisation might fall short of both legal requirements and professional codes of conduct.

7.2 Future Enhancements

  • Allow “what‑if” simulations (e.g., encrypting all sensitive data) to show how risk metrics improve.
  • Extend the dataset with simulated breach events and incident‑response timelines.
  • Integrate additional regulations (e.g., local data‑protection laws) into the checks.
  • Export dashboard summaries as PDF reports for management or audit teams.

Overall, the project reinforces a central lesson of Professional Issues in Computing: good engineering is not only about making systems work, but also about making them trustworthy, lawful and respectful of the people whose data they handle.
